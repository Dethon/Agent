## Invariants

- **The satellite is the Wyoming SERVER; the hub dials in** (default `--listen 0.0.0.0:10700`). A new hub connection supersedes the previous one (abort + await) so a dead-peer TCP wedge can't hold the exclusive `plughw` mic for the ~15-min retransmission timeout. The three ONNX models are parsed + optimized ONCE at boot (`WakeModels::load`, fail-fast) and shared across connections â€” re-arm after a reconnect is instant.
- **Cancellation safety**: hub/mic reads AND playback writes/drains are multi-await compound I/O, NOT `select!`-safe; they run in dedicated pump tasks (hub, mic, playback) feeding bounded mpsc channels, and the main `select!` only races `recv()` futures.
- **Playback pump**: the pump task is the single owner of the playback device. `audio-stop`'s drain (~0.5-2 s of buffered TTS) happens inside the pump, so wake/button/mic stay live during the reply tail; drain completions return on an unbounded channel (bounded would AB-deadlock) carrying a generation that gates the LED Idle/Listening transition (a stale completion can't blank a newer stream); playback errors stay connection-fatal; cues route through the pump too, so a cue player can never EBUSY-race a reply for the exclusive device (cues are dropped while a stream is active).
- **Keep-warm** (`--keep-warm`, Jabra unit only): the Jabra's playback DAC spins up cold on **each fresh `aplay` session** and glitches its first ~100 ms â€” inaudible inside a long TTS reply but ruinous for short cue beeps AND audible at the reply/chime onset. Crucially the glitch is a per-session-OPEN transient: a warm DAC does **not** survive a session close+reopen (verified on-device â€” keep-warm running, yet a fresh per-cue sink still glitched), so the only clean path is to play *inside an already-running stream*. So on the keep-warm path the pump holds **one** player open for the whole connection: while idle it tops it up one 20 ms silence chunk per 20 ms tick (paced at playback rate â†’ buffered-ahead stays ~aplay start-delay â‰ˆ 100 ms), and **everything â€” cues AND TTS replies/chimes â€” is INJECTED into this same running stream** (`write_pcm`, never a fresh open), so onsets are clean with **no added latency** (same ~100 ms a fresh sink had; no prime). Reply drain on this path is **estimated** (buffered-bytes / byte-rate) and only drives the LED â€” which is a no-op on the Jabra (no LED) and the satellite reports nothing back to the hub on drain, so precision is non-critical. The **non-keep-warm path is unchanged**: a fresh deep-buffer sink per stream with the exact `finish()`/`wait()` drain â€” LED devices (reSpeaker) run `--no-keep-warm` and keep it. Best-effort: a failed/dead keep-warm stream logs a warn and falls back to fresh per-stream/cue sinks (never worse than off); the silence top-up runs sequentially, never raced in `select!`. Proven on-device: cold beep choppy; beep/onset played inside the running stream clean â€” plain silence suffices (no signal/dither); the two satellite cues were fixed first (v2), then the reply onset + hub continuation-chime by injecting TTS too (v3). Off by default; reSpeaker-class devices pass `--no-keep-warm`.
- **Audio contract**: mic = 16 kHz mono S16LE in 1280-sample/80 ms chunks (arecord subprocess; bytes end-to-end internally, decoded to i16 only at the detector); playback sink = FIXED 22 050 Hz mono S16LE (aplay) that ignores announced rates â€” everything it plays must be 22 050 Hz: hub-side TTS and chime plus the embedded cue WAVs.
- **ALSA latency flags**: the default commands carry `arecord â€¦ -F 20000` (20 ms periods; the alsa-utils default of buffer/4 = 125 ms delayed every mic sample on the wake and STT paths) and `aplay â€¦ --start-delay=100000 -F 50000` (start at ~100 ms queued instead of the full-500 ms-buffer default; buffer stays 500 ms for underrun headroom). Keep them when overriding devices â€” EXCEPT the deployed Jabra unit drops `-F 20000` from the mic command: a small capture period was never confirmed working once the device is warm, and only the default ~125 ms period is proven on it, so reliability wins over the latency trim there. Plain-argv audio commands exec directly (no `sh -c`), so kill/supersede SIGKILLs aplay/arecord themselves; shell-shaped commands (WSL gain pipe) still go through sh.
- **Zero-lag pre-roll**: while idle, mic chunks fill a pre-roll ring (`--preroll-ms`, default 1000); a wake trigger flushes only the detection gap (3 chunks â‰ˆ 240 ms), never the wake word itself; a button press flushes the full ring.
- **LED**: the state machine publishes `LedState` (Idle/Listening/Thinking/Speaking) on a tokio watch channel; a per-connection render task owns the backend (`--led-gpio` pin or `--led-spi` for the ReSpeaker HAT APA102s on `/dev/spidev0.1`); Idleâ†’off, everything elseâ†’steady on, 120 s Thinking fallback mirroring the hub reply timeout; missing/failing LED hardware is never fatal. Idle after a reply still means actual-playback-complete (drain-completion-driven).
- **reSpeaker XVF3800 USB (deployed fran-office unit, MiniAmp speaker)**: 16 kHz-native S16LE both directions, 2ch (both capture channels carry the same processed mic signal â€” `plughw` stereoâ†’mono averaging is fine). Its capture engine only runs while its OWN playback stream is active (both UAC endpoints Synchronous off one internal clock): capture-alone opens EIO on read and a live capture dies the instant playback stops. Provisioning detects this (capture-alone probe) and installs `nabu-micclock.service` â€” an endless zero-stream into its unconnected speaker out that keeps capture clocked 24/7 â€” and passes `--no-wake-playback` (a wake tone adds nothing; the feeder owns clocking). The verdict is STICKY across re-provisions: the probe is only valid on a COLD engine (after the feeder has run, the engine stays clocked for minutes, capture-alone succeeds, and a re-probe would wrongly tear the feeder down â€” observed on-device 2026-07-16); a hardware swap needs `sudo rm /etc/systemd/system/nabu-micclock.service` first. Speaker output goes to the HiFiBerry MiniAmp (`dtoverlay=hifiberry-dac`, card `sndrpihifiberry`) via PipeWire; WirePlumber rules exclude the reSpeaker card and disable the unused HDMI/jack cards so the HAT is deterministically the only/default sink. Note on AEC (experiment 2026-07-16, NEGATIVE result â€” don't redo without new firmware/hardware): with playback on the MiniAmp the XVF3800's echo cancellation stays inert even when fed a proper far-end reference. A PipeWire loopback (virtual `speaker_frontend` default sink â†’ undelayed copy to a standalone `api.alsa.pcm.sink` on the Array + a filter-chain delay stage 20-120 ms â†’ MiniAmp, all in one `pipewire -c` client so targets exist at load) delivered a clean, clip-free, level-balanced reference verified at the DSP taps (`xvf_host` mux cat 4/12 = far-end pre/post gain, cat 3 = mic into SHF, cat 7 + `AEC_ASROUTONOFF 0` = linear residual): linear ERLE stayed 0.0 dB across the whole delay sweep while `AEC_AECCONVERGED=1`. Vendor guidance assumes the speaker hangs off the XVF3800's OWN 3.5 mm out (`AUDIO_MGR_SYS_DELAY` trims only Â±64..256 samples); a fully external DAC is outside the firmware's echo model. Real AEC would need the speaker driven from the XVF3800's jack (analog amp instead of MiniAmp) or new firmware. Gotchas worth keeping: the Array's UAC `PCM` mixer ships at âˆ’23/âˆ’20 dB as deliberate headroom for the Ã—8 `AUDIO_MGR_REF_GAIN` â€” setting it to 0 dB hard-clips the reference (now stored at âˆ’13/âˆ’20 dB via alsactl, clip-free); a filter-chain sink's monitor taps the POST-graph signal; `xvf_host` + libs live at `/opt/xvf3800-host/` on the Pi.
- **Defaults target a Jabra Speak2** â€” provisioning auto-detects the USB capture card by NAME (`arecord -l`; the id varies by variant: 75â†’J75, 55 MSâ†’MS, 55 UCâ†’UC) and addresses it as `plughw:CARD=<name>,DEV=0` on both commands. The capture side is **16 kHz mono native** (exactly the satellite's mic format â€” no resampling); `plughw` resamples only the 22050 Hz playback to a rate the device lists (48000/44100/32000/16000/8000). **The Jabra's mic ADC firmware-sleeps when idle**: capture-only opens return EIO, and from DEEP sleep (after a host reboot) it ignores even *silent* playback â€” only **real audio signal** wakes it (no `arecord` retries and no HID off-hook do; physical touch or a tone does). So the satellite **plays a brief TONE through the speaker on a cold mic open** (`--wake-playback-ms`, default unit value 500 ms; `play_wake_tone` synthesizes a 440 Hz sine, NOT zeros), opening the mic FIRST so an already-awake device never beeps, and retrying tone+open up to 20Ã— until capture streams a chunk; once a stream is held open continuously (the pre-roll path) it stays awake. (Index-pinning via `snd_usb_audio index=0` was removed â€” it collided with the Pi's built-in vc4-hdmi + headphone cards occupying ALSA slots 0-2, failing the USB probe with -16 so no card was created. An `/etc/asound.conf` 48 kHz plug device was also tried and removed â€” the capture is 16 kHz-ONLY, so a 48 kHz slave is "no configurations available".) No button/LED â€” the Jabra's buttons/LEDs are HID-telephony, unusable on Linux. The ReSpeaker 2-Mic HAT is the override path: `plughw:CARD=seeed2micvoicec,DEV=0` on both audio commands plus `--button-gpio 17 --led-spi` (needs `dtparam=spi=on` and the `spi` group); it doesn't sleep, so `--no-wake-playback` is fine.
- **Wire format**: frames are encoded as one contiguous buffer with event `data` sent once as the `data_length` body (the hub's reader prefers the body; its writer emits the same shape) â€” pinned by a codec test.

## Build & Deploy

`scripts/build-release.sh` cross-compiles via cargo-zigbuild + zig (the `zigcc-fp16-shim.sh` CC shim rewrites tract-linalg's `+fp16` -march feature to zig's `+fullfp16`) â€” never run bare `cargo zigbuild` for releases. `.cargo/config.toml` pins `-C target-cpu=cortex-a53 -C target-feature=-aes,-sha2` for the musl target (the Pi's silicon lacks the crypto extensions LLVM's cortex-a53 def would enable). Repo-root `scripts/provision-satellite-rs.sh <user@host> [mic-device]` installs the binary plus the templated `deploy/nabu-satellite.service` unit on a Pi (only dependency: `alsa-utils`; the unit pins the `performance` governor and `Nice=-10`) and, when no mic device is passed, auto-detects the USB card by name (addressing it `plughw:CARD=<name>,DEV=0`) plus a USB-autosuspend-off udev rule keyed on the device's vendor:product. I2S DAC/amp HAT units (MiniAmp) declare `DAC_OVERLAY=hifiberry-dac` â€” I2S is electrically undiscoverable, so the script writes the dtoverlay to config.txt, reboots, waits for the card, and continues (one-shot on a fresh Pi). Music units route the satellite's own playback (TTS + cues) through a `tts` ALSA softvol (control `TTS` on the speaker card, set to `TTS_VOLUME`%, default 65, re-asserted per provision) so agent-voice loudness is calibrated independently of music â€” the volume knob for amp HATs that have none (MiniAmp); tune live with `amixer -c <card> sset TTS <pct>%` + `sudo alsactl store`. The unit passes `--wake-playback-ms 500` for the Jabra's firmware mic-sleep (play-to-wake). qemu-emulation smoke tests need `--no-wake` (qemu's fp16 hwcaps activate tract f16 kernels that crash under emulation; a real A53 selects the f32 kernels). On-device E2E validation is still open, blocked on hardware â€” it should also read the `RUST_LOG=debug` per-chunk "wake inference" timing line.

## Running on WSL

Repo-root `scripts/wsl-satellite.sh` builds (`cargo build --release`, native target) and runs a satellite on the WSL host through WSLg PulseAudio; the dockerized hub dials `tcp://host.docker.internal:$SAT_PORT` (defaults match `McpChannelVoice/appsettings.Development.json`: 10700 = fran-office-01, 10600 = laura-office-01). Env knobs: `SAT_PORT`, `THRESHOLD` (wake threshold, default 0.5), `MIC_GAIN` (default 3.0 via a python `audioop` pipe â€” WSLg's mic bridge is quiet and the binary has no gain flag; needs python â‰¤ 3.12), `RUST_LOG`. `paplay --latency-msec=50` is mandatory on WSLg (the RDP sink's default buffer adds ~1.6 s playback latency, so the hub opens the wake-free follow-up window â€” 400 ms playback-tail echo guard, then 7 s window â€” while the reply is still audibly playing). Stale instances are detected via an `ss` LISTEN-state check because WSL2 mirrored networking makes loopback connects to dead ports hang instead of refusing.

**WSLg's RDP audio bridge audibly degrades playback** (harsh/crackly; the Linux-side chain measures bit-clean at the Pulse monitor tap â€” the corruption is in the RDPSinkâ†’Windows leg). Repo-root `scripts/wsl-satellite-winaudio.sh` is the clean-audio variant: same satellite in WSL, but the audio commands are Windows binaries run through WSL interop â€” `ffmpeg.exe` dshow mic capture (`-audio_buffer_size 50`, the dshow analogue of arecord's `-F 20000`) and `ffplay.exe` WASAPI playback (`-af adelay=150:all=1` because every fresh playback session can randomly glitch its first instants; a 120-180 ms earcon lives entirely inside that window, so the pad moves the artifact into silence). Needs the gyan.dev ffmpeg-release-essentials zip extracted to `%LOCALAPPDATA%\nabu-satellite\`.
